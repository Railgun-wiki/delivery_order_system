# 三种外卖订单系统架构对比与分析

| 维度 / 架构 | 1. 纯内存容器 (Deque + List) | 2. 嵌入式本地数据库 (SQLite) | 3. 分布式高可用 (PostgreSQL + MQ + Redis) |
| :--- | :--- | :--- | :--- |
| **持久化与宕机恢复** | **无**（重启即进程状态清空，丢数据） | **有**（基于单文件存储） | **强**（完整的主从复制与刷盘策略） |
| **并发访问性** | **弱**（只适用于单进程、单线程或GIL内锁死）| **中**（多线程下支持有限并发读，写容易受锁阻塞） | **极高**（消息队列削峰，分布式锁防止竞争） |
| **可扩展性 / 集群化**| **无**（只能独立部署） | **无**（多个应用实例如不在同一文件系统则无法协作） | **极高**（可无限扩展微服务节点去消费和处理订单） |
| **查询能力 / 报表分析**| **非常弱**（需要 Python 层面的线性遍历/过滤，性能差） | **良好**（能利用 SQL 与 Indexing 高效聚合统计，比如查餐厅/导出） | **极强**（支持复杂的联表查询、大数据分析） |
| **部署维护复杂度** | **极低**（0 依赖，插拔即用） | **极低**（仅单文件，随应用分发） | **极高**（需要维护整套消息中间件及缓存集群容灾） |

---

## 架构选择指导

### 结构一：纯内存容器 (`collections.deque` + `list`)

**选择场景**：
*   **刷算法题 / 面试验证 / 玩具 Demo**。
*   嵌入式小型硬件、一次性脚本里充当极轻量级调度器。

**主要操作**：
*   直接利用内置类型的 O(1) 特性，如 `deque.append()`、`deque.popleft()` 等来完成排队和弹出。
*   使用数组栈结构 `list.append()` 和 `list.pop()` 处理撤销和提交机制。

**为什么这样选**：
*   代码短小、执行效率极高（纳秒级内存操作）。没有额外依赖环境，最适合学习数据结构设计。

---

### 结构二：嵌入式本地数据库 (`SQLite`) —— 当前方案代码状态

**选择场景**：
*   **小型本地客户端应用 / 单体单店 SaaS 系统 / POS 机客户端**。
*   需求量每天千百单，不需要后台几十个服务协作，只需要在本地记账和报表导出。

**主要操作**：
*   利用 SQL 来控制一切操作：
    *   下单/等待/取消：`INSERT INTO ... VALUES (..., 'WAITING')` 及 `UPDATE ... SET status = 'CANCELLED'`
    *   获取下一单：基于状态 `SELECT ... LIMIT 1`（数据库表本身充当持久化队伍）。
    *   过滤与查询：直接将条件转译给 RDBMS，利用底层 B-Tree 索引查询。

**为什么这样选**：
*   **获得了完整的 ACID 事务保证和可靠数据持久化，但避免了复杂的微服务基建**。即使电脑断电强关，系统重启后订单不会丢。拥有数据库完整的查询和排序能力，使得后续业务扩展（按餐厅查询、统计排队数、导出所有）极其简单。

---

### 结构三：分布式高并发架构 (`PostgreSQL` + `MQ` + `Redis`)

**选择场景**：
*   **真正的美团 / 饿了么工业级生产线上**。
*   极高的全国规模 QPS、复杂的补贴与支付分布式事务。

**主要操作**：
*   `MQ` 挡在前头做异步事件：`mq.publish({"order_id": 1})`，隔离下单冲击。
*   `Redis` 用作告诉中间件缓存和无锁协调（Distributed Lock/List），维持系统秒级状态。
*   `PostgreSQL` 单纯做主数据落库和高价值财务数据的持久存储中心。

**为什么这样选**：
*   这是为了解决**横向扩容**和**物理可用性**的问题。当一个机房的服务挂了，其它节点可以接管；巨大的峰值通过消息队列削峰化解，系统不会因为超载而宕机。各个组件职责解耦，虽然增加运维复杂度但属于“海量业务必须承受之痛”。
